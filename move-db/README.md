# move-db

SF6全キャラの技データ(ダメージ・補正値・フレーム等)を持つデータベース。
[damage-calc](../damage-calc/README.md) が毎回技データを聞き直さずに済むようにするのが目的。
各キャラのデータは `moves/<char>.ts` に `satisfies CharacterMoveDb` 付きで置く(TSにすることで
[schema.ts](schema.ts) に対する型検査が効く)。

## 現状(2026-07-24時点)

全30キャラの公式フレーム表をTSVに落とし済み(`raw-frame-tsv/`、2361技)。ここから
`tools/tsv-to-ts.ts` で機械変換し、手作業パスで TODO を潰していく方針。

**ルーク([moves/luke.ts](moves/luke.ts))が1体目**。機械パス + id付与 + 派生関係
(`derivesFrom`/`derivesInto`)+ 無敵/空中/アーマーの構造化まで完了。残りは実機調査待ちの
項目(`hitstun`/`blockstun`/`hitCount`確定値/多段`hits`)と`notes`整理。他29キャラは未着手。

## スキーマ(暫定)

定義は [schema.ts](schema.ts) にある。

- `baseDamage` / `starterScaling` / `comboScaling` / `immediateScaling` / `minGuarantee` は
  [damage-calc](../damage-calc/damage-calc.ts) の `MoveInput` にそのまま渡せる形にする
- 値がない/未確認の項目は `null`

## TSVからTSへ移すときの方針

```bash
node --experimental-strip-types tools/tsv-to-ts.ts raw-frame-tsv/luke.tsv > moves/luke.ts
```

生成後は `moves/<char>.ts` を手作業パスで直接編集する(以降スクリプトの再生成はしない。
手編集が上書きされるため)。型検査は `npm run typecheck`。

**フレーム表から機械的に埋める項目**(`tools/tsv-to-ts.ts` が担当。判断が要る項目は
`null`/placeholderで残し `// TODO` を出す)

- `totalDamage` — 表のダメージ欄をそのまま。多段技でも段ごとに割らない
- `active` — 持続の長さ。`13-31`なら`31 - 13 + 1`
- `totalFrame` — 全体フレーム。ジャンプ攻撃は着地が終了条件なので全体フレームを持たず、
  代わりに`landingRecovery`(着地隙)に入れる
- `hitCount` — **暫定値**として`持続内訳`の区間数を入れる(内訳が無ければ1)。正確な値は実機調査で後から修正する
- `overAllProperties` — 表記載の属性。段ごとに属性が変わる多段技は`null`にして`hits[].properties`を使う
- `notation` — テンキー表記に変換(↓=2 等)。`強P強K`→`DI`・`中P中K`→`DP` のボタン対置換あり
- `guage`(Dゲージ増減/SA増加)、`recovery`、硬直差、`cancelableBy`(表の C/SA/SA2/SA3 をそのまま)

**実機調査で埋める項目**(表に無い)

- 飛び道具の`hitstun`/`blockstun`
- `cancelWindow` / `cancelRushWindow`
- `hitCount`の確定値、多段技の`hits`(段ごとのダメージ・属性・ゲージ)
- `recovery` は多段技だと表の値と変わるため要再調査

**手作業で埋める項目**

- `id` — スラグ規約は [id・notation規約](#idnotation規約) 参照
- `derivesFrom` / `derivesInto` — 派生関係。数が少ないので手作業(`window`は表に明記があるものだけ)
- 無敵/空中/アーマー — 備考から `invincibleFrames`/`airborneFrames`/`superArmorFrames` に構造化
  (`完全無敵`=4属性全部。判定の一部だけ無敵になるものは構造化せず`notes`止まり)

## id・notation規約

- **通常技/特殊技の`id`** — テンキー式 + 英字ボタン(`5LP`, `2MK`, `jHP`)。派生入力は `>` で繋ぐ(`6HP>HP`)
- **必殺技の`id`** — `強度(L/M/H/OD) + 名前`。強度を持たない派生技は名前のみ(`DDT`, `slamDunk`)
- **SAの`id`** — 名前のみ。CAは `paleRiderCA`
- **名前** — 公式英語版フレーム表(`raw-frame-tsv/<char>-en.tsv`)の技名を英訳せず camelCase 化。
  前置語(強度・OD等)の後の語は先頭大文字(`LRisingUppercut`, `ODNoChaser`)。日本語通称とズレる点に注意
  (ホールド→Charged, ジャスト→Perfect, エアリアル=Aerial, ステップ→Dash, ジャストパリィ→Perfect Parry)
- **投げの条件`近距離で`は無視して`condition: null`**(例外: 距離で技が変わるザンギエフ半回転等は保持)

**例外は汎化しない**

段ごとに違う補正がかかる技(キャミィのスイングコンビネーション、ケンのOD龍尾脚と
【奮迅脚】龍尾脚)、初段が空中の相手にしか当たらない判定、キャミィのTCが持つ2箇所の
キャンセルタイミングは、いずれもスキーマを拡張せず`notes`に自由記述で書く。
例外は数が少ないので、1件ずつ埋めていく。

**注釈記号`※`の対応先は`※`の有無では決まらない**

硬直欄の`※`に対応する備考項目に`※`が付いていないことがある(エドモンド本田のしゃがみ強P)。
どの列の注釈かは、列と備考の文面を突き合わせて判断する。

## フレーム表HTMLのパース

公式サイトのフレーム表を、加工しやすいTSV/CSVに落とすパーサーが
[tools/parse-frame-html.ts](tools/parse-frame-html.ts) にある。

```bash
node --experimental-strip-types tools/parse-frame-html.ts raw-frame-html/luke.html > raw-frame-tsv/luke.tsv
node --experimental-strip-types tools/parse-frame-html.ts raw-frame-html/luke.html --csv
```

入力のHTMLは`raw-frame-html/`に、出力のTSVは`raw-frame-tsv/`に置く。公式サイトはSPAでフレーム表がJS描画のため、
URLを直接fetchしても表は取れない。ブラウザで
`https://www.streetfighter.com/6/character/<キャラ>/frame` を開き、DevToolsのElementsで
`<html>`のouterHTMLをコピーして保存する。日本語版・英語版のどちらでも動く。

出力は次の20列。**値の解釈は一切しない**。`※11`・`全体 47`・`22+着地後12`のような表記は
セルの文字列のまま出力する。数値化やスキーマへの写像は後段の別スクリプトの担当。

| 列                                      | 備考                                                           |
| --------------------------------------- | -------------------------------------------------------------- |
| キャラ / カテゴリ                       | カテゴリは表のセクション見出し(通常技・必殺技等)               |
| 技名 / 条件 / 入力                      | 公式表では1セル。条件は`（ジャンプ中に）`等の先頭のカッコ書き  |
| 発生 / 持続 / 持続内訳 / 硬直           | 持続内訳は多段技のみ(ツールチップに隠れているヒットごとの内訳) |
| ヒット硬直差 / ガード硬直差             |                                                                |
| キャンセル / ダメージ / コンボ補正値    |                                                                |
| Dゲージ増加ヒット / 減少ガード / 減少PC |                                                                |
| SAゲージ増加 / 属性 / 備考              |                                                                |

補足:

- 入力はアイコン画像なのでファイル名からテキストに戻している(`↓↘→ + 中P`、
  派生は`⇒`で繋ぐ)。レバーの`→`と紛らわしいので派生の矢印は別の字にしてある
- 列の対応付けはtdのCSSクラス名で行うため、列の順番が変わっても壊れない。ただし
  クラス名自体(`frame_startup_frame`等)が変わると壊れる
- セル内に複数項目がある場合(備考・コンボ補正値)は`|`で連結する
- 未知の入力アイコンが出てきたら`[ファイル名]`として出力し、stderrに警告を出す

## 対象キャラ

全キャラ。データ量が多いのでキャラ単位で区切って進める想定。

TSVは全30キャラ分ある。ヤスミンのみ配信前でフレーム表が存在しないため対象外。
TS化はルークが着手済み(実機項目を除き完了)、他29キャラは未着手。

## 保留事項

- **飛び道具のhitstun/blockstun(絶対値)**: 発生・持続のフレームが弾の着弾タイミングと一致しないため、
  格闘ゲーム標準の「全体フレーム-持続終了フレーム=硬直」という単純計算が使えない。実機調査で埋める。
- **多段技(1入力で複数ヒットする技)**: `hits`配列で表現できる設計にした(旧保留事項は解消)。
  ターゲットコンボ(ダブルインパクト等)は`derivesFrom`/`derivesInto`で繋いだ別々の技として
  表現する。
- **硬直欄に残る例外**: `全体 N`と`着地後N`を除くと、次の4形が残る(全60件)。
  - `※N`(24件) — 数値は普通の硬直。`※`の中身は`recovery.whiff`/`.block`に落ちるもの
    (`※空振り時硬直3F増加`)と、特殊状態での値を示すもの(春麗4技の`※行雲流水を継続した場合の数値`)がある
  - `全体 ※N`(3件) — 動作の分岐で全体フレームが変わる(A.K.I.の蛇軽功、ガイルのソニックブレイク)。
    `totalFrame`単数では両方入らない
  - `着地N`(2件) — `着地後N`の表記揺れ。`landingRecovery`に寄せる
  - `74※107`(1件) — ジェイミーのSA1武麗禽のみ。未確認
- **属性・キャンセル欄の表記揺れ**: TSVの素の値には次の揺れがある。JSON化時に解く必要がある。
  - 属性: `・`で1段内の複合(`上・弾`)、連結で段ごとの属性(`上中下中`)、`/`で条件分岐
    (`上・弾/上・空弾`)、先頭`※`で備考への注釈(`※中`)
  - キャンセル: 注釈の`※`が前置・後置・カンマ区切りのいずれもあり、位置に意味の違いはない。
    条件付きキャンセルか否かは備考の文面を読むしかない
- **飛び道具の行分割**: 公式表では弾を出す動作の行と弾そのものの行が分かれている
  (ディージェイのエアスラッシャー、ブランカのブランカちゃん爆弾等)。動作側の行は
  ダメージ0・発生なしになる。ディージェイの弱版のようにフェイントで弾が出ない技もある。
